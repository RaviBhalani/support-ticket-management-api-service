## Domain & Core Features

The system manages customer support tickets. Key ticket fields:

- `customer_name`, `customer_email`, `subject`, `description`, `priority`, `category`
- `status` — enum: `OPEN`, `IN_PROGRESS`, `RESOLVED`, `CLOSED`

REST API operations:
- **Create ticket** — `POST /tickets`
- **Get ticket by ID** — `GET /tickets/{id}`
- **List tickets** — `GET /tickets` with pagination and filtering by `status`, `priority`, `category`
- **Update ticket status** — `PATCH /tickets/{id}/status` (must validate state transitions)

## Architecture Expectations

The assignment expects a layered architecture with clear separation of concerns:

- **Schemas/DTOs** — request and response models with validation (Pydantic)
- **Service layer** — business logic, state transition validation
- **Repository layer** — all database access isolated here
- **Enums** — for `status`, `priority`, `category`
- **Config** — environment-based configuration (`.env` file)
- **Logging** — structured logging throughout

## Database Design

PostgreSQL schema with:
- `tickets` table — core ticket data with audit timestamps (`created_at`, `updated_at`)
- `ticket_events` table — history of status changes and processing results
- Proper indexes and constraints; enums at DB level where appropriate
- Migrations via Alembic; seed scripts for quick testing
